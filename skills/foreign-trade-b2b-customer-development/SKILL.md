## 目标

用这个 Skill 帮外贸人把“找客户”做得更像人工开发流程，而不是让模型凭印象乱猜。默认围绕 8 步走：

```text
候选发现 → 官网确认 → 品类验证 → 客户类型判断 → 联系方式收集 → 判断依据 → 开发建议 → 表格输出
```

输出时也要围绕这 8 步组织。名称要像外贸业务表，不要像技术报告或侦查报告。

## 适用场景

适用于传统外贸 B2B 客户开发，例如：

- 按产品找进口商、批发商、经销商、品牌商、零售商、工程商、OEM/ODM 买家。
- 按国家/地区开发市场。
- 按行业、应用场景、竞品、展会名单、已有客户名单做筛选和补全。
- 根据官网、公开页面、目录、PDF、社媒公开页、企业目录等信息判断客户是否值得开发。
- 整理联系人、邮箱、联系页面、开发建议和可跟进表格。

## 任务入口分类

先把用户请求归到一种或多种入口：

1. `单个客户分析`：用户给公司名、官网或社媒公开页。
2. `按产品开发客户`：用户给产品、材料、用途、配件、替代品或应用场景。
3. `按关键词开发客户`：用户给关键词、搜索词或市场词。
4. `按行业开发客户`：用户给行业、上下游、渠道或垂直场景。
5. `按国家/地区开发客户`：用户给国家、区域、语言市场或客户类型。
6. `补全已有客户名单`：用户给 Excel、CSV、公司名单、官网列表或混合文本。
7. `按竞品开发客户`：用户给竞争品牌、同行官网、竞品型号或竞品渠道。
8. `按展会/协会/目录开发客户`：用户给展会名单、协会名单、黄页目录或采购商名单。

只问最少的下一步问题。能开始就先开始，不要让用户填很长表。

## 工作边界

只使用公开、可访问的信息。

- 不登录用户账号，不读取 Cookie、密码、私钥、API Key 或本地隐私文件。
- 不绕过验证码、登录、付费墙、反爬、平台风控或权限限制。
- 不自动批量抓取 Google、LinkedIn、Facebook、Instagram 等平台的受限结果。
- 不编造公司、官网、邮箱、联系人、采购意图、进口记录或认证信息。
- 信息不足时写：`未找到`、`未查询`、`打不开`、`需要人工确认`、`用户提供`、`推测，未验证`。

## 工具能力核查：有工具必须用，没工具不能装作查过

这个 Skill 不绑定某个具体工具或插件，而是让 Codex、Hermes 或其他 Agent 按“当前可用工具能力”自己调用上游工具完成。不要让用户手动替 Agent 调工具；只有当前环境确实没有能力、页面被限制或资料不足时，才请用户补充资料。

正式开发前先核查这些能力，并在本轮结果中留下记录：

```text
网页搜索：可用/不可用/未显示
网页读取：可用/不可用/未显示
浏览器/动态页面：可用/不可用/未显示
文档/PDF读取：可用/不可用/未显示
表格/文件处理：可用/不可用/未显示
公司/域名/邮箱辅助工具：可用/不可用/未显示
社媒公开页读取：可用/不可用/未显示
地图公开信息查询：可用/不可用/未显示
```

门禁规则：

- 没有网页搜索工具，也没有用户给的候选名单/搜索结果时，不得生成“真实客户名单”；只能输出关键词、搜索式和需要用户补充的材料。
- 没有读取官网、官网文字或用户提供官网截图时，不得写 `官网已确认`。
- 没有产品页、分类页、目录、案例、PDF 或用户提供的公开材料时，不得写 `品类已确认`。
- 没有 Contact 页面、公开邮箱页面或用户提供的联系方式来源时，联系方式状态写 `未找到`、`未查询` 或 `需要人工确认`。
- 每个进入 `客户开发总表` 的客户，都必须尝试查询社媒公开页和 Google Maps/公开地图信息；查不到、打不开、工具不可用或页面受限时，也要在主表写明状态和原因。
- 搜索摘要、模型记忆、公司名相似度只能用于候选发现，不能作为官网确认、品类验证或联系方式依据。
- 每个重要判断都要能回到一个公开来源、用户提供材料或明确的“未查询原因”。

## 运行模式判断

正式开发前，根据当前工具能力和用户提供材料，先判断本轮属于哪种模式，并告诉用户。

### A. 自动开发模式（推荐）

当前 Agent 至少具备：

```text
网页搜索 + 网页读取 + 表格/文件输出
```

使用方式：可以正常从零开发客户，按完整流程执行候选发现、官网确认、品类验证、联系方式收集、社媒公开页和 Google Maps/公开地图查询、表格输出。

### B. 半自动整理模式

当前 Agent 缺少网页搜索，但用户提供了以下任一材料：

```text
公司名单
官网列表
搜索结果复制内容
展会名单
协会/目录名单
Excel/CSV
PDF目录
网页截图或网页文字
```

使用方式：不从零发现客户，重点做筛选、官网确认、品类验证、联系方式整理、社媒/地图公开信息尝试查询和表格输出。

### C. 策略模式

当前 Agent 没有网页搜索，也没有网页读取，用户也没有提供名单或网页资料。

使用方式：不能生成真实客户名单，只能输出关键词、Google 搜索式、目标客户类型、优先国家/地区、推荐开发路径和需要补充的资料。

最低工具要求：

- 从零开发客户最低需要：`网页搜索 + 网页读取 + 表格/文件输出`。
- 缺少网页搜索：不能从零发现客户。
- 缺少网页读取：不能做官网确认和品类验证。
- 缺少表格/文件输出：可以先输出 Markdown/CSV 文本，但要说明文件输出受限。


## 工具路由：让 Agent 自己调用上游工具

按任务阶段选择工具能力，不强制某个工具名称：

1. **候选发现**：优先用网页搜索工具；搜索词要组合产品 + 客户类型 + 国家/语言，不要只搜一个大词。
2. **官网确认**：优先用网页读取工具读取官网、About、Products、Brands、Contact、Store/Dealer/Distributor 页面。
3. **动态页面**：只有已知公开页面读取不到时，才用浏览器/动态网页工具打开；只读取可见公开内容。
4. **品类验证**：优先读取产品分类页、解决方案页、公开 catalogue/PDF、案例页。
5. **公司背景**：公司信息工具只能辅助补充总部、行业、规模和业务类型，不能替代官网判断。
6. **官网身份**：域名/官网判断工具用于辅助判断域名与公司是否匹配。
7. **邮箱验证**：只验证公开页面找到或用户提供的邮箱；不要自动生成采购邮箱。若推测，必须标注 `推测，未验证`。
8. **社媒公开页查询**：每个客户都要尝试查询官网外链或公开搜索结果中的 LinkedIn、Facebook、Instagram、YouTube、X/Twitter、TikTok 等公开公司页；受限即停止，并记录状态。
9. **Google Maps/公开地图查询**：每个客户都要尝试查询 Google Maps 或公开地图/门店页中的公司地址、电话、营业状态、评价概况或门店信息；受限即停止，并记录状态。
10. **表格输出**：优先用表格/文件工具或本 Skill 的脚本整理 CSV/Excel；没有时输出 Markdown 表格或 UTF-8 CSV 字段内容。

如果当前环境没有对应工具，先判断运行模式，不要一上来让用户补一堆材料。自动开发模式继续开发；半自动整理模式基于用户提供材料整理；策略模式只输出搜索策略和最少补充材料选项。不要把未查询内容写成已查询。

## 社媒公开页和地图查询：每个客户都要尝试

社媒公开页和 Google Maps/公开地图不是主判断依据，但属于客户开发的强制检查项。每个进入 `客户开发总表` 的客户，都要尝试查询并记录结果。

只检查公开可访问的信息。不登录、不读取 Cookie、不绕过验证码、不绕过平台风控、不访问私密内容、不批量抓取受限结果。页面要求登录、出现验证码、触发风控、频率限制或工具无法访问时，立即停止该来源，并把状态写进主表。

必查尝试顺序：

1. 先看官网页脚、Contact、About、News 页面是否有官方社媒链接。
2. 再用公司名 + 国家/城市 + 平台名查询公开公司页。
3. 再尝试 Google Maps/公开地图，核对地址、电话、营业状态、门店/办公室信息。
4. 找不到不编造，受限不绕过；写清楚 `未找到`、`页面受限`、`工具不可用` 或 `需要人工确认`。

平台处理：

- LinkedIn：辅助判断公司定位、行业、公开员工角色、招聘和业务方向；要求登录或受限时停止。
- Facebook：查看公开公司页、About、门店、帖子里的业务活动和联系方式。
- Instagram：查看公开品牌风格、产品展示、零售/消费品信号，不作为主要联系方式来源。
- YouTube：查看公开产品演示、展会视频、应用案例、设备运行视频。
- Reddit：了解市场痛点、竞品反馈、应用讨论；不把 Reddit 帖子当成客户名单依据。
- X/Twitter：查看公开公司动态、展会、行业活动；优先级低于官网和目录。
- TikTok：只在公开可访问或用户提供链接/截图时使用；不绕过登录或风控。
- Google Maps/公开地图：核对公司地址、电话、营业状态、门店/办公室、公开评价概况；不抓取受限内容，不绕过 Google 风控。

`客户开发总表` 必须体现每个客户的社媒和地图查询情况，并体现资料完整度。推荐至少包含：`社媒公开页查询情况`、`社媒公开页链接`、`Google Maps查询情况`、`Google Maps链接/信息`、`社媒/地图未查询原因`、`查询完整度`、`未确认项`、`未查询原因`、`下一步补齐`。

## 执行要透明，不能黑盒

每次正式开发前，先用 3-6 行告诉用户本轮怎么做，并说明本轮运行模式：

```text
本轮模式：自动开发模式/半自动整理模式/策略模式。
本轮我会按：候选发现 → 官网确认 → 品类验证 → 客户类型判断 → 联系方式收集 → 判断依据 → 开发建议 → 表格输出 来做。
目标数量：X 个。
优先客户类型：进口商/经销商/工程商/...
优先市场：国家/地区。
输出：客户开发总表 + 需要的辅助表。
```

执行时遵守：

- 每轮正式开发都要记录 `本轮查询说明`：用了哪些工具/来源、查了哪些页面、哪些来源没查、没查原因、对结果有什么影响、下一步怎么补齐。
- 每个客户都要尝试查询官网外链、社媒公开页和 Google Maps/公开地图；结果必须进入 `客户开发总表`。
- 工具可用时，由 Agent 自己调用工具完成对应步骤；不要让模型只凭记忆补全。
- 先找候选，再逐个确认，不要一边搜一边直接下结论。
- 每个客户必须有一行进入 `客户开发总表`，不要把核心信息拆散在多个表里。
- 最终只能有一个主表，名称固定为 `客户开发总表`。
- 辅助表只能补充过程，不能替代主表。
- 如果用户要求 30 个，最终少于 30 个，必须明确写：`目标 30 个，本轮完成 N 个，缺口 X 个，原因是...，下一步建议...`。
- 聊天预览只展示 5-10 行，完整结果放文件或完整表格。
- 预览里长 URL 用域名显示，完整链接放表格字段。
- 不输出星级、分数、排行榜；用业务判断。
- 不把两个“总表”同时交付给用户；如果有多个来源表，先合并成一个 `客户开发总表`。

## 数量完成度规则

用户给目标数量时，必须跟踪：

```text
目标数量
候选数量
完成官网确认数量
完成品类验证数量
进入客户开发总表数量
缺口数量
缺口原因
下一步补齐方式
```

默认规则：

- 用户没说数量：宽泛开发默认 30 个候选，深度整理默认 10-20 个。
- 用户说“开发 30 个客户”：尽量交付 30 个进入 `客户开发总表` 的客户。
- 工具或资料不够时，不要装作完成；要说明完成度和缺口。
- 不要用不合格客户凑数。宁可少，也要写清楚原因。

## 判断标准要写清楚

品类状态统一用：

```text
已确认：官网有明确产品页、分类页、目录、案例或解决方案。
部分确认：官网业务相关，但没有直接对应用户产品。
未确认：只看到模糊行业相关，没找到对应品类。
打不开：页面无法访问或工具无法读取。
```

客户类型建议拆成两个字段：

```text
主客户类型：进口商/经销商/批发商/工程商/品牌商/...
客户类型说明：为什么这样判断。
```

## 标准开发流程

### 1. 候选发现

先扩大候选，不要一开始就过度筛选。

要做：

- 明确产品、行业、国家/地区、客户类型和数量目标。
- 扩展英文和当地语言关键词。
- 找候选公司和官网。
- 记录为什么把它放进候选。

不要做：

- 只凭公司名或关键词就判断是目标客户。
- 把博客、媒体、黄页、平台店铺当成客户公司。

### 2. 官网确认

确认这个公司是否有官网，官网是否像官方站。

看这些信息：

- About / Company / Who we are
- Contact / Head office
- Products / Brands / Solutions
- Store / Dealer / Distributor / Where to buy
- 域名、公司名、Logo、地址、邮箱后缀是否互相对应

### 3. 品类验证

确认客户是否真的卖、用、采购或经营用户的产品相关品类。

要找：

- 产品分类页
- 品牌/产品线
- catalogue/PDF
- 应用场景
- 行业解决方案
- 展会或新品页面

如果只是关键词相似，但官网没有对应产品或应用，要写 `品类未确认`。

### 4. 客户类型判断

把客户分成外贸人能理解的类型：

- 进口商
- 经销商
- 批发商
- 零售商
- 品牌商
- OEM/ODM 买家
- 工程商/项目商
- 代理商
- 电商卖家
- 系统集成商
- 贸易商
- 暂不确定

判断要基于官网文字和业务模式，不要只靠模型猜。

### 5. 联系方式收集

只收集公开商业联系方式：

- 官网 Contact 页面
- 公开邮箱
- 公开电话
- 联系表单
- LinkedIn/Facebook/Instagram/YouTube 等公开公司页
- 公开采购、供应商、合作入口

不要混淆不同公司的联系方式。不要编造联系人。

### 6. 判断依据

每个重要判断都要有依据。这里统一叫“判断依据”，方便外贸用户理解。

至少记录：

- 官网依据
- 品类依据
- 客户类型依据
- 联系方式来源
- 需要人工确认的问题

### 7. 开发建议

用外贸业务语言给建议，不用打分。

建议标签：

- `建议开发`
- `先确认后开发`
- `暂不优先`
- `信息不足`

建议内容包括：

- 推荐切入产品
- 推荐开发角度
- 首封邮件应该怎么说
- 首封默认不带附件、不放图片、不放视频、不放多个链接
- 后续可以准备什么资料
- 还缺什么信息
- 是否需要人工复查

### 8. 表格输出

输出表格时，sheet 或章节名称要口语化、清楚、符合中国外贸用户心智。

优先使用这些标签：

```text
客户开发总表
候选发现
官网确认
品类验证
客户类型判断
联系方式收集
判断依据
开发建议
表格输出
本轮查询说明
开发信建议
```

命名要求：所有标签都要口语化、业务化，让外贸用户一眼知道这一页是干什么的。

无论有没有辅助表，最终都必须有且只有一个主表，名称固定为：`客户开发总表`。
